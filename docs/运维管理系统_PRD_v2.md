## 技术选型

### 0.1 架构概述

系统采用**前后端完全分离 + MCP 智能层**三层架构，后端提供 RESTful API，前端为独立 SPA 应用，MCP Server 将业务能力抽象为 AI Tool 供模型调用。通过 Nginx 反向代理统一对外提供服务。

```
                          ┌─────────────────────────────┐
                          │      AI 客户端 (外部)        │
                          │  Claude Desktop / CodeBuddy  │
                          └──────────────┬──────────────┘
                                         │ MCP Protocol
                                         │ (stdio / HTTP+SSE)
                          ┌──────────────▼──────────────┐
                          │    织云 MCP Server           │
                          │    (cloudweave-mcp)          │
                          │    Tool 注册 & 调用转发       │
                          └──────────────┬──────────────┘
                                         │ 内部 HTTP
┌─────────────────────────────────────────────────────────────┐
│                        Nginx (反向代理)                       │
│      /mcp/* → MCP Server   /api/* → 后端   /* → 前端静态资源  │
└──────────┬──────────────────────┬───────────────────────────┘
           │                      │
           ▼                      ▼
┌─────────────────────┐  ┌──────────────────────────┐
│   前端 (SPA)         │  │   后端 (Django REST)      │
│   Vue 3 + Vite      │  │   DRF + Celery            │
│   TypeScript        │  │   Python 3.12+            │
│   Pinia / Vue Router│  │                           │
└─────────────────────┘  └──────────┬────────────────┘
                                    │
                  ┌─────────────────┼─────────────────┐
                  ▼                 ▼                  ▼
           ┌──────────┐     ┌──────────┐      ┌──────────┐
           │PostgreSQL│     │  Redis   │      │  Celery  │
           │  16      │     │ 缓存/队列 │      │  Worker  │

```

### 0.5 选型理由

**为什么前后端分离？**
- 后端只负责数据和业务逻辑，前端独立迭代 UI 交互，互不阻塞
- 拓扑可视化（vis-network）、图表（ECharts）等重前端功能更适合 SPA 架构
- 未来可扩展移动端或小程序，复用同一套 API
- 前后端各自独立测试、独立部署，CI/CD 流水线分工明确

**为什么 Vue 3 而非 React？**
- Vue 3 Composition API + `<script setup>` 语法简洁，学习曲线平缓，适合团队快速上手
- Element Plus 提供开箱即用的中后台组件（Table/Form/Dialog/抽屉），显著减少重复造轮子
- 与 Django 社区的 Vue-DRF 组合方案成熟，生态配套完善

**为什么选择 PostgreSQL？**
- JSONB 类型支持索引和高效查询，远优于 MySQL 的 JSON 存储，非常适合 `extra_data`、`condition_config` 等半结构化字段
- 原生支持 `ArrayField`、`HStoreField`、`DateRangeField` 等 Django 高级字段类型，无需额外插件
- 窗口函数、CTE（WITH 子句）在处理关联追溯和趋势分析等复杂查询时表达能力更强
- Django ORM 对 PostgreSQL 的支持最为完整（`django.contrib.postgres` 提供专属字段和索引）
- 严格的 SQL 标准遵循和 ACID 特性，适合需要审计追溯的运维数据场景

**为什么 vis-network 而非 D3.js / AntV G6？**
- vis-network 配置式 API 简洁，30 行代码即可渲染力导向图
- 物理引擎稳定，500 节点以内性能表现优秀
- ECharts 补充图表需求，与 vis-network 各司其职

**为什么选择 MCP 协议暴露 AI 能力？**
- MCP 是 Anthropic 推出的开放标准协议，已被 Claude Desktop、CodeBuddy、Cursor、Continue 等主流 AI 工具原生支持，无需自建对话前端即可接入 AI
- Tool 定义使用 JSON Schema 标准化，AI 模型可自动理解参数含义并生成正确的调用参数
- stdio 传输适合本地开发与个人使用（零网络配置），HTTP+SSE 传输适合团队共享部署
- MCP Server 作为独立进程运行，与 Django 后端解耦，通过内部 HTTP 调用复用现有 DRF Service 层，不侵入核心业务代码
- API Key 权限模型可精确控制哪些 AI 客户端只能查询、哪些可以执行写操作，安全性可控
- 生态兼容性好：同一套 MCP Tool 可同时被 Claude、GPT、Gemini 等多个模型调用，不绑定特定 AI 供应商

## 一、产品概述

### 1.1 产品定位

面向集团公司/中小型团队的综合资产运维管理平台，以**资源关联拓扑**为核心设计理念，覆盖两大资产管理域——**服务器资产**（服务器/域名/IP/端口/项目）与**办公资产**（电脑/手机/SIM卡/平板）的统一登记、关联追踪、流程管控与可视化展示。

### 1.2 核心价值

- **"一图胜千言"**：通过拓扑视图从任意资源节点出发，直观追溯其上下游完整关联链路，告别"这个 IP 是谁的？这个域名解析到哪台机器？"的运维黑洞。
- **"关联即真相"**：建立项目 → 服务器 → IP → 域名 → 端口的全链路模型，任何一个节点下线或变更时，系统可快速评估影响范围。
- **"合规可追溯"**：ICP 备案全生命周期管理，备案主体变更留痕，方便合规审计。
- **"API 驱动运维"**：对接主流云平台 OpenAPI，自动同步服务器资产、账户余额，余额不足或资源到期时主动告警，减少人工巡检成本。
- **"AI 自然交互"**：通过 MCP (Model Context Protocol) 将运维能力暴露为标准化 AI Tool，支持自然语言查询与操作——"帮我查一下 api.example.com 解析到哪些服务器"、"列出余额不足 1000 元的云账号"。

### 1.3 目标用户

| 角色 | 职责 | 关注点 |
|---|---|---|
| 运维工程师 | 日常资源管理与维护 | 资源登记、关联配置、拓扑探索、云 API 同步 |
| 项目经理/技术负责人 | 项目资源盘点 | 项目关联的资源总览、成本归属 |
| 财务/成本管理员 | 云账户成本管控 | 余额监控、费用趋势、余额告警 |
| 合规/安全管理员 | 备案合规审计 | ICP 备案状态、备案变更历史 |
| IT 资产管理员 | 办公设备全生命周期管理 | 资产登记、借用审批、调拨流转、手机卡实名合规 |
| 普通员工 | 办公资产借用 | 发起借用申请、查看个人借用记录 |
| 系统管理员 | 平台运维 | 用户权限、操作日志 |
| AI 助手使用者 | 自然语言运维交互 | 通过 MCP 客户端（CodeBuddy/Claude Desktop 等）用自然语言查询资产、追溯链路、查看告警 |

### 2.1 资产管理模块

**目标**：对项目、服务器、域名、IP、端口五类核心资源进行 CRUD 管理，每类资源作为独立实体存在。

#### 2.1.1 项目管理

| 功能点 | 说明 |
|---|---|
| 项目列表 | 分页展示、支持按名称/状态/负责人搜索筛选 |
| 项目详情 | 展示项目基本信息 + 关联的服务器/域名/端口汇总卡片 |
| 新增/编辑项目 | 名称、描述、负责人、状态（运行中/已下线/维护中）、所属团队/部门 |
| 删除项目 | 软删除，保留历史关联记录 |
| 资源概览卡片 | 在详情页以卡片形式展示该项目下关联的服务器数、域名数、端口数 |

#### 2.1.2 服务器管理

| 功能点 | 说明 |
|---|---|
| 服务器列表 | 展示主机名、IP、云平台、操作系统、状态，支持多条件筛选 |
| 服务器详情 | 基本信息 + 关联项目列表 + 网络接口（多IP） + 开放端口列表 + 部署的域名 |
| 新增/编辑服务器 | 主机名、操作系统、CPU/内存/磁盘规格、SSH 端口、登录方式、所属云平台、到期时间、备注 |
| 批量导入 | 支持 CSV/Excel 批量导入服务器信息 |
| 状态标记 | 运行中 / 已关机 / 已销毁 / 维护中 |

#### 2.1.3 域名管理

| 功能点 | 说明 |
|---|---|
| 域名列表 | 展示域名、注册商、ICP备案状态、解析IP数、关联项目、到期时间 |
| 域名详情 | 基本信息 + 解析记录（A记录/CNAME等）+ 关联项目 + ICP备案信息卡片 + 变更历史时间线 |
| 新增/编辑域名 | 域名、注册商、注册日期、到期日期、DNS 服务商、备注 |

                    │   项目    │
                    │ Project  │
                    └────┬─────┘
                         │ 部署于 (deploy_on)
                         │ 多对多
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         ┌─────────┐ ┌─────────┐ ┌─────────┐
         │ 服务器A  │ │ 服务器B  │ │ 服务器C  │
         │ Server  │ │ Server  │ │ Server  │
         └────┬────┘ └────┬────┘ └────┬────┘
              │ 拥有 (has_ip)         │
              │ 一对多                │
              ▼                      ▼
         ┌─────────┐           ┌─────────┐
         │  IP 1   │           │  IP 2   │
         │(公网)   │           │(内网)   │
         └────┬────┘           └─────────┘
              │ 解析至 (resolves_to)
              │ 多对多
              ▼
         ┌─────────┐
         │ 域名 A   │
         │ Domain  │
         └────┬────┘
              │ 关联项目 (related_project)
              │ 多对多
              ▼
         ┌──────────┐
         │   项目    │
         │ Project  │
         └──────────┘

端口 (Port) 依附于 服务器 + IP 的组合上
         ┌─────────┐
         │ 服务器   │─── 开放端口 (opens_port) ───▶  Port (8080/TCP/nginx)
         │ Server  │─── 开放端口 (opens_port) ───▶  Port (3306/TCP/mysql)
         └─────────┘
```

#### 2.2.1-附2 微服务部署架构模型

当多个项目之间存在服务间调用关系时（如前端项目需要调用后端项目 API），系统通过**服务依赖**与**内部域名解析**两个机制建模跨项目的通信拓扑。

**场景一：多节点分布式部署（前后端分离）**

```
┌──────────────────────────────────────────────────────────────────┐
│  项目 G (前端服务)                     项目 C (后端 API 服务)       │
│  role: app_server                     role: app_server           │
│                                      role: database (可选)        │
│                                                                  │
│  ┌──────────┐  ┌──────────┐         ┌──────────┐  ┌──────────┐  │
│  │ 服务器 E  │  │ 服务器 F  │         │ 服务器 A  │  │ 服务器 B  │  │
│  │ Port:3000│  │ Port:3000│         │ Port:8080│  │ Port:8080│  │
│  │ (Vite)   │  │ (Vite)   │         │ (FastAPI)│  │ (FastAPI)│  │
│  └────┬─────┘  └────┬─────┘         └────┬─────┘  └────┬─────┘  │
│       │              │                   │              │        │
│       └──────┬───────┘                   └──────┬───────┘        │
│              │                                  │                │
│              │      service_dependency          │                │
│              │  ┌─────────────────────┐         │                │
│              └─▶│ 项目G 依赖 项目C      │◀────────┘                │
│                 │ 内部域名:            │                          │
│                 │ api.project-c.internal│                         │
│                 │ 协议: HTTP, 目标端口:8080│                      │
│                 └──────────┬──────────┘                          │
│                            │                                     │
│                            ▼                                     │
│                 ┌──────────────────────┐                         │
│                 │  内部 DNS 解析        │                         │
│                 │  api.project-c.internal│                       │
│                 │        ↓              │                         │
│                 │  ┌──────────────────┐ │                         │
│                 │  │ VIP: 10.0.1.100  │ │  ← 内部VIP，高可用      │
│                 │  │ (server_vip)     │ │                         │
│                 │  └────────┬─────────┘ │                         │
│                 │           │           │                         │
│                 │  ┌────────┴────────┐  │                         │
│                 │  ▼                 ▼  │                         │
│                 │ 服务器A(master)  服务器B(backup)                 │
│                 └──────────────────────┘                         │
│                                                                  │
│  外部访问链路：                                                     │
│  域名 www.xxx.com → 公网IP → Nginx(服务器D) → 前端(服务器E,F)      │
└──────────────────────────────────────────────────────────────────┘
```

**关键设计要素**：

| 要素 | 说明 |
|---|---|
| 服务依赖 | 项目 G（前端）依赖项目 C（后端 API）。通过 `service_dependency` 表记录 |
| 内部域名 | `api.project-c.internal` 是仅内网可解析的内部域名，前端通过此域名调用后端 |
| 内部 VIP | 后端 A、B 服务器共享一个内部 VIP（10.0.1.100），前端请求 VIP 时自动负载均衡到 A 或 B |
| 高可用 | VIP + 双后端节点，任一后端宕机不影响服务；前端 E、F 双节点，Nginx 层做负载均衡 |
| 动态扩容 | 新增后端服务器 B' 时，将其加入 `project_server`(项目C) 和 `server_vip`(VIP节点)，无需修改前端配置 |

**场景二：单节点部署（前后端合设）**

```
┌──────────────────────────────────────────────┐
│  服务器 X                                     │
│                                              │
│  ┌──────────────────────────────────────┐    │
│  │  项目 G (前端)    Port: 3000          │    │
│  │  Vite dev server / Nginx 静态托管     │    │
│  └──────────┬───────────────────────────┘    │
│             │ service_dependency              │
│             │ 内部域名:                        │
│             │ api.project-c.internal          │
│             ▼                                  │
│  ┌──────────────────────────────────────┐    │
│  │  项目 C (后端)    Port: 8080          │    │
│  │  FastAPI / Django                    │    │
│  └──────────────────────────────────────┘    │
│                                              │
│  域名解析（单点）：                             │
│  www.xxx.com → IP(服务器X公网) → Nginx:443    │
│  api.project-c.internal → 127.0.0.1:8080     │
│      (或直接 localhost 调用)                   │
│                                              │
│  Nginx 配置（同机）：                          │
│  / → proxy_pass http://127.0.0.1:3000 (前端) │
│  /api/ → proxy_pass http://127.0.0.1:8080    │
│                                              │
│  project_server 记录：                        │
│  · 项目C → 服务器X (role: app_server)        │
│  · 项目G → 服务器X (role: app_server)        │
│  server_proxy_mapping：                      │
│  · 服务器X(Nginx):443 → 服务器X(前端):3000   │
│  · 服务器X(Nginx):443 → 服务器X(后端):8080   │
└──────────────────────────────────────────────┘
```

**两种模式对比**：

| 维度 | 多节点分布式 | 单节点合设 |
|---|---|---|
| 服务器数量 | 后端 2+台 + 前端 2+台 + 代理 1+台 | 1 台 |
| 高可用 | VIP + 多节点，无单点故障 | 单点，宕机即不可用 |
| 内部 DNS | 解析至内部 VIP | 解析至 127.0.0.1 或宿主机内网 IP |
| 扩容方式 | 新增节点加入 project_server + server_vip | 垂直扩容（升级服务器规格） |
| 适用场景 | 生产环境、高并发业务 | 开发测试、内部工具、低流量业务 |
| `project_server` | 每个项目多条记录（每个节点一条） | 每个项目一条记录（同一服务器） |
| `service_dependency` | 记录项目间依赖 + 内部域名 | 同左，但内部域名指向 localhost |

**模型扩展性**：通过 `service_dependency` + `server_vip` + `server_proxy_mapping` 的组合，系统可表达从单机到大规模微服务的任意部署拓扑，包括：
- 微服务网格（Service Mesh）：多个项目形成依赖 DAG，每个项目多节点部署
- 读写分离：database 角色节点区分读写（role: database_read / database_write）
- 多环境隔离：通过 `project_server.environment` 区分 production / staging，内部域名也按环境命名（`api.project-c.staging.internal`）

#### 2.6.3 数据级权限

除功能级权限外，系统支持按**主体公司**和**项目**两个维度进行数据隔离：

| 权限维度 | 配置方式 | 效果 |
|---|---|---|
| 主体公司隔离 | 用户绑定可访问的主体公司列表 | 该用户仅能看到绑定主体下的资产、域名、云账号 |

**默认规则**：

- `employee`：仅可查看自己借用/领用的设备 + 公开的项目信息
- `readonly`：可查看已授权的数据范围（由管理员配置）

#### 2.7.3 服务器资产同步

| 功能点 | 说明 |
|---|---|
| 实例拉取 | 调用云平台 DescribeInstances / DescribeInstances 等 API，拉取指定地域下的所有实例 |
| 字段映射 | 将云平台返回字段映射到系统 Server 模型：实例 ID → instance_id、实例名称 → hostname、操作系统 → os、CPU/内存 → cpu_cores/memory_gb、公网 IP → 自动创建 IP 记录、状态 → status、到期时间 → expire_date、可用区 → 存入 extra_data JSON 字段 |

| 同步日志 | 每次同步生成详细日志：同步时间、耗时、新增/更新/标记下线数量、失败项及错误原因 |

|---|---|
| 规则列表 | 展示所有告警规则：规则名称、告警类型、告警级别、启用状态、上次触发时间 |
| 新增/编辑规则 | 规则名称、告警类型（余额阈值 / 资源到期 / 同步失败）、告警目标（指定云账号或全部）、触发条件、告警级别（严重/警告/提醒）、通知渠道、通知对象 |
| 余额告警规则 | 条件配置：当可用余额 ≤ X 元时触发告警，支持设置多级阈值（如余额 < 1000 元「警告」、余额 < 100 元「严重」）|
| 资源到期告警规则 | 条件配置：服务器/域名到期前 N 天触发告警（默认 30/15/7 天），可按云账号分组设置不同阈值 |
| 同步失败告警规则 | 条件配置：连续 N 次同步失败（或失败率超过 X%）时触发告警，防止凭证过期导致长期数据失真 |
| 规则启用/停用 | 支持临时关闭某条告警规则而不删除 |
| 通知渠道配置 | 每条规则可独立配置通知渠道：站内消息（默认开启）、邮件、企业微信（需预先配置 Webhook）|

#### 2.7.6 告警通知与处理

| 功能点 | 说明 |
|---|---|
| 告警记录列表 | 展示历史告警记录：告警标题、级别、触发时间、状态（已触发/已确认/已解决）|
| 告警详情 | 展示告警详情：触发条件、当前值 vs 阈值、涉及资源链接 |
| 告警确认 | 运维人员可"确认"告警（表示已知悉），记录确认人和时间 |
| 告警解决 | 问题处理后可标记为"已解决"，支持填写处理备注 |
| 告警收敛 | 同一规则在冷却期内（默认 2 小时）不重复发送通知，避免告警轰炸 |
| 通知模板 | 邮件/企业微信通知使用模板化格式：告警标题 + 详情摘要 + 操作链接 |
| 免打扰时段 | 支持配置免打扰时间窗口（如 22:00-08:00），时间段内告警不在即时渠道推送，仅记录待次日提醒 |

### 2.8 AI 智能集成模块 (MCP)

**目标**：通过 MCP (Model Context Protocol) 协议，将织云的全部业务能力抽象为标准化的 AI Tool，使 AI 模型（如 Claude、GPT 等）可通过自然语言直接与运维系统交互——查询资产、追溯链路、查看告警、获取余额，无需打开 Web 界面。

#### 2.8.1 MCP 架构概述

```

                          │       AI 客户端              │
                          │  (CodeBuddy / Claude Desktop │
                          │   / Cursor / Continue 等)     │

                          │  └───────────┬────────────┘  │

                          │  ┌───────────▼────────────┐  │
                          │  │  Tool Implementations   │  │

#### 2.8.2 MCP Tool 清单设计

将织云全部 API 能力按业务域抽象为 MCP Tool，每个 Tool 有明确的 `name`、`description`（供 AI 模型理解用途）、`inputSchema`（JSON Schema 参数定义）。共设计 **7 个域、30+ Tool**：

**域一：资源查询 (Resource Query)**

| Tool Name | 描述 | 核心参数 |
|---|---|---|
| `search_resources` | 全局模糊搜索，跨项目/服务器/域名/IP/端口 | `query: string`, `resource_types: string[]`, `limit: int` |
| `list_projects` | 分页查询项目列表 | `status: string`, `owner: string`, `page: int`, `page_size: int` |
| `get_project_detail` | 获取项目详情及其关联资源汇总 | `project_id: int` 或 `project_name: string` |
| `list_servers` | 分页查询服务器列表 | `cloud_platform_id: int`, `status: string`, `page: int` |
| `get_server_detail` | 获取服务器详情（含IP、端口、关联项目、域名） | `server_id: int` 或 `hostname: string` |
| `list_domains` | 分页查询域名列表 | `icp_status: string`, `registrar_id: int`, `expiring_days: int` |
| `get_domain_detail` | 获取域名详情（含解析记录、备案信息、关联项目） | `domain_id: int` 或 `domain_name: string` |
| `list_ips` | 分页查询 IP 列表 | `ip_type: string`, `server_id: int` |
| `list_ports` | 分页查询端口列表 | `server_id: int`, `service_name: string` |

**域二：关联追溯 (Relationship Tracing)**

| Tool Name | 描述 | 核心参数 |
|---|---|---|
| `trace_upstream` | 从指定资源向上游追溯（如域名 → IP → 服务器 → 项目） | `resource_type: string`(project/server/domain/ip/icp_entity), `resource_id: int` |
| `trace_downstream` | 从指定资源向下游追溯（如项目 → 服务器 → IP → 域名 → 端口） | `resource_type: string`, `resource_id: int` |
| `trace_full_chain` | 从指定资源双向追溯完整关联链路 | `resource_type: string`, `resource_id: int` |
| `get_impact_analysis` | 模拟某资源变更/下线的影响范围分析 | `resource_type: string`, `resource_id: int`, `change_type: string` |

**域三：云平台 & 余额 (Cloud & Balance)**

| Tool Name | 描述 | 核心参数 |
|---|---|---|
| `list_cloud_accounts` | 列出所有云平台账号 | — |
| `get_cloud_account_detail` | 获取云账号详情（含余额、同步状态） | `account_id: int` |
| `get_balance` | 查询指定云账号当前余额 | `account_id: int` |
| `get_balance_history` | 查询余额历史趋势 | `account_id: int`, `days: int` |
| `get_sync_status` | 查询最近同步状态 | `account_id: int` |
| `trigger_sync` | 手动触发云平台资产同步 | `account_id: int`, `sync_services: string[]` |

**域四：告警 (Alerts)**

| Tool Name | 描述 | 核心参数 |
|---|---|---|
| `list_alerts` | 查询告警记录，支持按级别/状态/时间筛选 | `severity: string`, `status: string`, `days: int` |
| `get_alert_summary` | 告警统计概览（各级别数量、未处理数） | — |
| `acknowledge_alert` | 确认告警 | `alert_id: int`, `note: string` |
| `resolve_alert` | 标记告警已解决 | `alert_id: int`, `note: string` |
| `list_alert_rules` | 查询告警规则列表 | `is_enabled: bool` |

**域五：ICP 备案 (ICP)**

| Tool Name | 描述 | 核心参数 |
|---|---|---|
| `get_icp_status` | 查询域名备案状态 | `domain_name: string` |
| `get_icp_detail` | 获取备案详细信息 | `domain_name: string` |
| `get_icp_change_history` | 查询备案变更历史 | `domain_name: string` |

**域六：统计 & 概览 (Dashboard)**

| Tool Name | 描述 | 核心参数 |
|---|---|---|
| `get_dashboard_summary` | 全局概览：资源数量统计、告警数、到期资源数 | — |
| `get_expiring_resources` | 列出即将到期的服务器/域名 | `days: int` |
| `get_resource_stats` | 按云平台/项目等维度统计资源分布 | `group_by: string` |

**域七：拓扑 (Topology)**

| Tool Name | 描述 | 核心参数 |
|---|---|---|
| `get_topology_data` | 获取指定范围的拓扑图数据（节点+边 JSON） | `project_id: int`, `resource_types: string[]` |
| `get_topology_neighbors` | 获取某节点的一度关联节点 | `resource_type: string`, `resource_id: int` |

### 2.9 办公资产管理模块

**目标**：面向集团公司多主体场景，对办公电脑、手机、SIM 卡、平板等设备进行统一登记、全生命周期追踪与流程化管控。将服务器资产（IT 基础设施）与办公资产（员工设备）纳入同一平台，实现"一个平台，两类资产"的统管模式。

#### 2.9.1 资产类型管理

办公资产区别于服务器资产的显著特征是**品类多样且各具专属属性**，因此需要灵活的资产类型定义机制：

| 功能点 | 说明 |
|---|---|
| 资产类型列表 | 展示所有定义的资产类型：名称、图标、启用状态、关联资产数 |
| 预置类型 | 系统预置：台式机、笔记本、办公手机、手机卡(SIM)、平板、显示器、打印机、网络设备（交换机/路由器） |
| 自定义类型 | 管理员可新增资产类型（如"U 盾"、"监控摄像头"），设置名称和图标 |
| 自定义字段 | 每种资产类型支持添加额外字段：字段名、字段类型（文本/数字/日期/下拉选择/布尔）、是否必填 |
| 字段模板 | 预置类型自带推荐字段模板（如"笔记本"自动包含：品牌、型号、序列号、CPU、内存、硬盘、操作系统），管理员可增删字段 |
| 启用/归档 | 不再使用的资产类型可归档，归档后不可新增该类资产 |

**预置字段示例**：

| 资产类型 | 预置专属字段 |
|---|---|
| 台式机/笔记本 | 品牌、型号、序列号(S/N)、CPU、内存(GB)、硬盘(GB/SSD/HDD)、操作系统、MAC 地址、采购日期、保修到期 |
| 办公手机 | 品牌、型号、IMEI、操作系统(iOS/Android)、存储(GB)、采购日期、保修到期 |
| 手机卡(SIM) | 手机号、运营商(移动/联通/电信)、套餐名称、月租费、ICCID、开卡日期→关联实名信息子表 |
| 平板 | 品牌、型号、序列号、IMEI（蜂窝版）、存储(GB)、操作系统、采购日期、保修到期 |
| 显示器 | 品牌、型号、尺寸(英寸)、分辨率、接口类型 |

#### 2.9.2 主体公司管理（全系统通用组织架构）

主体公司是织云系统中**跨业务模块的通用实体**，在导航中独立入口"主体公司"统一管理。办公资产管理、域名归属、云平台账号归属、ICP 备案主体、项目归属均共享此组织架构。办公资产上下文中的使用方式如下：

| 功能点 | 说明 |
|---|---|
| 主体列表 | 展示集团下所有主体公司：名称、简称、统一社会信用代码、法人代表、状态 |
| 新增/编辑主体 | 公司全称、简称、统一社会信用代码、法人代表、注册资本、成立日期、经营范围、注册地址、联系人，支持父子级层级 |
| 主体详情 — 关联资源总览 | 点击主体名称进入详情页，跨模块展示该主体关联的全部资源：项目列表、云平台账号列表、域名列表、ICP 备案域名、办公资产统计（各类型数量+状态分布） |
| 主体树 | 支持父子级组织架构（集团 → 子公司 → 孙公司），便于权限隔离和数据归集 |
| 资产归属 | 每项办公资产必须归属到一个具体主体；域名、云账号、项目亦可关联主体 |
| 主体间调拨 | 支持将资产从主体A调拨至主体B，记录调拨原因、审批人、调拨时间 |

#### 2.9.3 办公资产登记

| 功能点 | 说明 |
|---|---|
| 资产列表 | 分页展示：资产编号（自动生成）、资产名称、类型、品牌型号、所属主体、当前使用人、状态、位置 |
| 资产详情 | 基本信息 + 专属字段 + 使用记录时间线（采购→领用→借用→归还→调拨→报废）+ 关联的员工/项目 |
| 新增资产 | 选择资产类型 → 自动加载该类型的专属字段表单 → 填写通用信息（资产名称、所属主体、存放位置、采购信息）→ 填写专属字段 |
| 快速录入 | 支持批量录入同类资产（如一次性入库 10 台同型号笔记本），填写一次字段模板后批量生成 |
| 二维码标签 | 每项资产支持生成唯一二维码标签（含资产编号和基本信息 URL），打印后粘贴至设备，扫码即可查看详情/发起借用 |
| 资产状态 | 在库 / 已领用 / 借用中 / 调拨中 / 维修中 / 已报废 |

#### 2.9.4 手机卡实名制管理

| 功能点 | 说明 |
|---|---|
| 实名信息登记 | SIM 卡资产详情页的专属子表：实名人姓名、身份证号（脱敏存储，仅显示前 6 位 + 后 4 位）、身份证有效期、实名日期 |

### 3.2 核心实体表设计

#### 3.2.0 主体公司表 (company_entity) — 全系统通用

主体公司是织云系统中**跨业务场景的通用核心实体**，统一承载域名归属、云平台账号归属、办公资产归属、ICP 备案主体、项目归属等所有需要"归属于哪个公司/法人实体"的场景。各业务模块通过 ForeignKey 关联此表，实现各场景独立管理、统一查询。

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | BigAutoField(PK) | 是 | 主键 |
| name | CharField(200) | 是 | 公司全称，如"XX科技有限公司"，唯一 |
| short_name | CharField(50) | 否 | 简称，如"子公司A" |
| credit_code | CharField(50) | 否 | 统一社会信用代码（18 位），不为空时唯一 |
| legal_person | CharField(50) | 否 | 法定代表人姓名 |
| registered_capital | DecimalField(14,2) | 否 | 注册资本（万元） |
| established_date | DateField | 否 | 成立日期 |
| business_scope | TextField | 否 | 经营范围 |
| address | TextField | 否 | 注册地址 |
| contact_person | CharField(50) | 否 | 日常联系人 |
| contact_phone | CharField(20) | 否 | 联系电话 |
| contact_email | EmailField | 否 | 联系邮箱 |
| parent | ForeignKey('self') | 否 | 上级主体（集团→子公司→孙公司层级） |
| status | CharField(20) | 是 | active / inactive |
| remark | TextField | 否 | 备注 |
| created_at | DateTimeField | 是 | 创建时间 |
| updated_at | DateTimeField | 是 | 更新时间 |

**唯一约束**：name 唯一；credit_code 不为空时唯一。

**多场景关联汇总**（通过各业务表 ForeignKey 反向查询）：

| 业务场景 | 关联方式 | 说明 |
|---|---|---|
| 项目归属 | `Project.company → FK(CompanyEntity)` | 项目归属于哪个主体公司 |
| 云平台账号归属 | `CloudPlatform.company → FK(CompanyEntity)` | 云账号归属哪个主体（费用归属） |
| 域名归属 | `Domain.company → FK(CompanyEntity)` | 域名以哪个主体名义注册 |
| ICP 备案主体 | `IcpRecord.company → FK(CompanyEntity)` | 备案时填写的企业主体，替代原先的纯文本字段 |
| 域名注册商 | `Registrar.company → FK(CompanyEntity)` | 注册商账户归属哪个主体 |
| 办公资产归属 | `OfficeAsset.company → FK(CompanyEntity)` | 办公设备归属哪个主体 |
| 服务器归属 | `Server.company → FK(CompanyEntity, nullable)` | 服务器资产的归属主体（可选，私有化部署场景） |

**设计原则**：
- 公司主体作为独立管理域，不隶属于任何一个业务模块。在导航中设独立入口"主体公司"，统一维护。
- 各业务模块创建资源时，若涉及"归属于哪家公司"，统一从此表下拉选择，确保数据一致性。
- 通过一个公司 ID，可跨模块查询其全部关联资源：旗下项目、云账号、域名及备案、办公设备。

#### 3.3.5 服务器代理映射表 (server_proxy_mapping)

精确记录 Nginx/HAProxy 等反向代理服务器与后端应用服务器的代理关系。这是解决"域名 → 代理服务器 → 后端服务器"关联链路的关键中间表。

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | BigAutoField(PK) | 是 | 主键 |
| proxy_server | ForeignKey(Server) | 是 | 代理服务器（部署 Nginx/HAProxy 的服务器） |
| backend_server | ForeignKey(Server) | 是 | 后端目标服务器 |
| backend_port | ForeignKey(Port) | 否 | 后端目标端口（可关联 Port 表或直接用 port_number） |
| backend_port_number | IntegerField | 否 | 后端端口号（当未在 Port 表中登记时使用） |
| proxy_port | IntegerField | 否 | 代理服务器监听端口（如 443/80） |
| server_name | CharField(255) | 否 | Nginx server_name 指令值（域名），如 `api.example.com` |
| proxy_path | CharField(500) | 否 | 代理路径，如 `/api/`（默认 `/` 表示全部流量） |
| protocol | CharField(10) | 是 | 代理协议：http / https / tcp / udp（四层代理场景） |
| project | ForeignKey(Project) | 否 | 关联项目（便于追溯该代理规则服务于哪个项目） |
| remark | TextField | 否 | 备注（如对应的 Nginx upstream 名称） |
| created_at | DateTimeField | 是 | 创建时间 |

**唯一约束**：(proxy_server, backend_server, proxy_port, server_name, proxy_path) 组合唯一。

**使用说明**：基于此表，系统可以——
- 从代理服务器出发，查询它代理了哪些后端服务器（`WHERE proxy_server_id = X`）
- 从后端服务器出发，查询哪些代理服务器将流量转发给它（`WHERE backend_server_id = X`）
- 从域名出发，通过 `server_name` 字段定位具体的代理规则，再追溯到后端

#### 3.3.6 服务依赖表 (service_dependency)

记录微服务架构中项目之间的调用依赖关系。前端项目通过内部域名调用后端项目 API 时，依赖此表描述调用拓扑。

| 字段名 | 类型 | 必填 | 说明 |

| id | BigAutoField(PK) | 是 | 主键 |
| from_project | ForeignKey(Project) | 是 | 调用方项目（如前端项目 G） |
| to_project | ForeignKey(Project) | 是 | 被调用方项目（如后端项目 C） |
| internal_domain | CharField(255) | 是 | 内部域名，如 `api.project-c.internal` |
| protocol | CharField(10) | 是 | 通信协议：http / https / grpc / tcp |
| target_port | IntegerField | 是 | 目标端口，如 8080 |
| target_path | CharField(500) | 否 | API 路径前缀，如 `/api/v1/` |
| environment | CharField(20) | 是 | 环境标签（与 project_server.environment 对应） |
| remark | TextField | 否 | 备注 |

## 五、关键业务逻辑

### 5.1 资源关联追溯算法

系统需要支持从任意资源节点出发，追溯其上下游关联链路。以下为各资源的追溯逻辑：

**从项目出发（下游追溯）**：
```
输入: 项目 ID
步骤:
  1. 查询 project_server 表 → 获取关联的服务器列表
  2. 对每台服务器：
     a. 查询 ip_address 表 (WHERE server_id = X) → 获取 IP 列表
     b. 查询 port 表 (WHERE server_id = X) → 获取端口列表
  3. 对每个 IP：
     a. 查询 domain_ip 表 (WHERE ip_id = X) → 获取域名列表
  4. 查询 project_domain 表 (WHERE project_id = 项目ID) → 获取直接关联的域名
  5. 汇总去重，构建链路树
```

**从域名出发（上游追溯）— 含代理穿透**：
```
输入: 域名 ID
步骤:
  1. 查询 domain_ip 表 → 获取解析目标 IP 列表
  2. 对每个 IP，按 ip_type 分两路处理：

     路径 A — 普通 IP（public/private/elastic）：
       a. 查询 ip_address.server → 获取归属服务器
       b. 该服务器如果在 project_server 中 role=reverse_proxy 或 load_balancer，
          则进一步查询 server_proxy_mapping (WHERE proxy_server_id = X) → 穿透到后端服务器列表
       c. 对每台穿透到的后端服务器，查询 project_server → 获取实际部署的项目

     路径 B — VIP IP（vip）：
       a. 查询 server_vip 表 (WHERE ip_id = X) → 获取所有挂载该 VIP 的服务器节点
       b. 对每台节点服务器，执行路径 A 的 b-c 步骤（穿透代理到后端）

  3. 查询 project_domain 表 → 获取域名直接关联的项目
  4. 查询 icp_record 表 → 获取备案信息
  5. 汇总去重，构建链路树
```

**从服务器出发（双向展开）— 含代理角色识别**：
```
输入: 服务器 ID
步骤:
  1. 查询 project_server 表 → 获取该服务器关联的项目及其 role
  2. 向上游：
     a. 查询 ip_address 表 → 获取该服务器的 IP 列表
     b. 对每个 IP，查询 domain_ip → 获取解析至该 IP 的域名
     c. 若 IP 为 vip 类型，查询 server_vip → 获取同 VIP 下的其他节点
  3. 向下游：
     a. 若 role 包含 reverse_proxy 或 load_balancer：
        查询 server_proxy_mapping (WHERE proxy_server_id = X) → 获取所有后端服务器
        对每台后端服务器，继续追溯其 project_server 和 port
     b. 若 role 包含 app_server：
        查询 server_proxy_mapping (WHERE backend_server_id = X) → 反查哪些代理服务器将流量转发给它
  4. 汇总去重，构建双向链路树
```

**从企业主体（ICP 备案主体）出发**：
```
输入: 企业主体名称 或 icp_record 中的 company_name
步骤:
  1. 查询 icp_record 表 (WHERE company_name = X AND status = 'active') → 获取该主体下所有备案记录
  2. 对每条备案记录：
     a. 获取关联域名 → domain 表
     b. 对每个域名，调用"从域名出发"的追溯算法 → 获取解析 IP、归属服务器、部署项目
  3. 汇总去重，按域名分组，构建以企业主体为根节点的链路树
  4. 额外信息：负责人、联系电话（来自 icp_record）
输出: 企业主体为根节点的多域名关联图谱
```

**核心原则**：以上所有追溯算法遵循同一模式——输入资源类型 + 资源 ID，沿关联关系表逐层展开，直至无更多关联或触及预设深度上限（默认 4 层）。新增资源类型（如企业主体）只需定义其在关联链中的位置和相邻表即可复用同一追溯引擎。

### 5.2 影响分析算法

当某资源状态发生变更（如服务器下线）时，系统需自动评估影响范围：

```
输入: 资源类型 & 资源 ID & 变更类型(下线/维护/删除)
步骤:
  1. 调用 5.1 中的追溯算法，获取该资源的完整上下游链路
  2. 按资源类型分类统计影响范围：
     - 影响的项目列表
     - 影响的域名列表
     - 影响的 IP 列表
     - 影响的端口列表
  3. 对影响范围内的每个资源，检查是否有冗余/备用方案：
     - 域名是否还有其他解析 IP？
     - 项目是否还部署在其他服务器上？
  4. 生成影响分析报告，标注高/中/低风险等级
输出: 影响分析报告 JSON
```

**风险等级定义**：
- **高风险**：资源下线后，有项目/域名完全没有备用方案，直接不可用
- **中风险**：资源下线后，有项目/域名的部分功能受影响，但有备用
- **低风险**：资源下线后，影响范围内的资源均有完整冗余

### 5.3 ICP 备案变更流程

```
变更流程状态机：
                ┌─────────┐
                │  draft  │ (草稿)
                └────┬────┘
                     │ 提交审核
                     ▼
                ┌─────────┐
         ┌──────│ pending │ (待审核)
         │      └────┬────┘
         │           │
         │    ┌──────┼──────┐
         │    ▼             ▼
         │ ┌──────────┐ ┌──────────┐
         │ │ rejected │ │ approved │ (通过)
         │ │ (驳回)    │ └────┬─────┘
         │ └────┬─────┘      │
         │      │            │ 自动更新 icp_record
         │      │ 重新提交    │ 记录变更历史
         │      ▼            │
         │ ┌─────────┐       │
         └─│ pending │◄──────┘ (可重新提交)
           └─────────┘
```

**审核规则**：
- 变更申请提交后，需由具有 `icp_review` 权限的用户审核
- 审核通过后系统自动更新 `icp_record` 对应字段，无需手动操作
- 审核驳回时需填写驳回原因，支持驳回后修改重新提交
- 同一域名的同一字段如果存在 `pending` 状态的变更，不允许再次提交该字段的变更

### 5.5 云平台 API 同步逻辑

#### 5.5.1 同步调度策略

系统通过 Celery Beat 实现定时同步调度，同时支持手动触发：

```
Celery Beat 定时调度
  │
  ├── 每分钟扫描 cloud_platform 表
  │     获取所有 sync_enabled=True 的账号
  │
  ├── 对每个账号：
  │     │ last_sync_at + sync_interval_minutes ≤ now ?
  │     │
  │     ├── 是 → 创建 sync_task 异步任务
  │     │
  │     └── 否 → 跳过
  │
  └── 手动触发：直接创建 sync_task，不受间隔限制
```

#### 5.5.2 资产同步流程（以腾讯云 CVM 为例）

```
输入: cloud_platform_id
步骤:
  1. 从 cloud_platform 表获取 API 凭证（SecretId/SecretKey）和 sync_regions
  2. 创建 sync_log 记录，状态为 running
  3. 对每个 sync_region：
     a. 调用 tencentcloud-sdk-python 的 DescribeInstances API
     b. 获取该地域所有 CVM 实例列表
     c. 对每个实例进行字段映射：
        - InstanceId → instance_id
        - InstanceName → hostname
        - OsName → os
        - CPU → cpu_cores
        - Memory → memory_gb
        - SystemDisk.DiskSize → disk_gb
        - PublicIpAddresses → 创建或更新 IP 记录（ip_type=public）
        - PrivateIpAddresses → 创建或更新 IP 记录（ip_type=private）
        - InstanceState → status 映射（RUNNING→running, STOPPED→stopped, TERMINATED→terminated）
        - ExpiredTime → expire_date
        - Placement.Zone → 存入 extra_data JSON
     d. 与本地 Server 表做 Diff（按 instance_id 匹配）
  4. 差异处理：
     - 新增实例（云端有、本地无）→ 创建 Server + IP 记录
     - 更新实例（云端有、本地有且属性变化）→ 更新 Server + 同步 IP
     - 已销毁实例（云端已不存在、本地状态非 terminated）→ 标记 status=terminated
  5. 如果 sync_services 包含 "billing"，执行余额同步（见 5.5.3）
  6. 更新 sync_log 状态为 success/failed/partial
  7. 更新 cloud_platform.last_sync_at 和 last_sync_status
输出: sync_log 记录
```

#### 5.5.3 余额同步流程

```
输入: cloud_platform_id
步骤:
  1. 从 cloud_platform 表获取 API 凭证
  2. 调用云平台 Billing/Account API 查询余额
     - 腾讯云：调用 DescribeAccountBalance API
     - 阿里云：调用 QueryAccountBalance API
  3. 解析返回结果：
     - 现金余额 (CashBalance)
     - 代金券余额 (VoucherBalance)
     - 信用额度 (CreditBalance)
     - 冻结金额 (FrozenAmount)
  4. 计算总余额 = 现金余额 + 代金券余额
  5. 写入 balance_record 表
  6. 触发余额告警评估（见 5.6）
输出: balance_record 记录
```

### 5.6 告警评估与通知逻辑

#### 5.6.1 告警评估流程

告警评估在以下时机触发：
- 余额同步完成后
- 每日定时任务（到期扫描）
- 同步任务完成后（失败检测）

```
告警评估引擎流程:
  1. 获取所有 is_enabled=True 的 alert_rule
  2. 按 alert_type 分类评估：

     【余额阈值告警】
     - 获取云账号最近一条 balance_record
     - 若 total_balance ≤ condition_config.threshold → 触发告警
     - 检查冷却期：若该规则在 cooldown_minutes 内已触发过 → 跳过

     【资源到期告警】
     - 扫描 server 表 WHERE expire_date - NOW() IN condition_config.days_before
     - 扫描 domain 表 WHERE expire_date - NOW() IN condition_config.days_before
     - 每条到期资源生成一条告警记录
     - 同一条规则+同一资源在冷却期内不重复触发

     【同步失败告警】
     - 查询 sync_log 表，统计该云账号最近 N 条的 status
     - 若连续 failure 次数 ≥ condition_config.consecutive_failures → 触发告警
     - 同步恢复成功后自动标记该类告警为 resolved

  3. 创建 alert_record，状态为 triggered
  4. 调用通知分发（见 5.6.2）
```

#### 5.6.2 通知分发逻辑

```
通知分发流程:
  1. 获取告警规则的 channels 配置和 notify_users 列表
  2. 检查当前时间是否在 quiet_start ~ quiet_end 免打扰时段内：
     - 是 → 仅写站内消息，邮件/企业微信延迟至 quiet_end 后发送
     - 否 → 立即按 channels 配置发送
  3. 各渠道通知方式：
     - 站内消息：写入 Django 内置通知表，前端轮询或 WebSocket 推送
     - 邮件：通过 Django send_mail + SMTP 发送格式化 HTML 邮件
     - 企业微信：调用 Webhook 推送 Markdown 格式消息卡片
  4. 记录通知发送日志（成功/失败），失败时重试 1 次
```

#### 5.6.3 告警收敛策略

| 收敛策略 | 说明 |
|---|---|
| 冷却期 | 同一规则在 cooldown_minutes 内不重复触发通知 |
| 资源去重 | 同一资源+同一规则在冷却期内仅产生一条告警记录 |
| 自动恢复 | 同步失败告警在下次同步成功后自动标记为 resolved |
| 日汇总 | 可选：每日 09:00 发送告警汇总报告（"今日共触发 3 条告警，1 条未处理"）|

### 5.5 数据一致性保障

| 场景 | 处理策略 |
|---|---|
| 删除服务器 | 级联标记关联的 IP、端口为"已失效"状态，保留历史关联记录 |
| 删除 IP | 自动解除 domain_ip 中的关联记录，记录操作日志 |
| 删除项目 | 解除与服务器、域名的关联，端口标记为"未关联项目" |
| 域名备案状态变更 | 如果域名从"已备案"变更为"备案注销"，对应的 icp_record 标记为 `cancelled` |

**软删除策略**：所有核心实体均采用软删除（`is_deleted=True`），数据不物理删除，确保关联关系可追溯。前端列表页默认过滤 `is_deleted=True` 的记录，管理员可在筛选条件中选择"显示已删除"。

## 六、非功能性需求

### 6.1 性能要求

| 指标 | 要求 |
|---|---|
| 页面加载时间 | 列表页 < 2s，详情页 < 1.5s |
| 拓扑图渲染 | 节点数 < 500 时，渲染时间 < 3s |
| API 响应时间 | P99 < 500ms |
| 并发用户数 | 支持 50+ 并发用户 |

### 6.2 安全要求

- 所有 API 接口需进行权限校验
- 敏感操作（删除、备案变更审核）需二次确认
- 操作日志不可删除
- 云平台 API 密钥（SecretId/SecretKey）采用 AES-256-CBC 加密存储于数据库，密钥加密密钥（KEK）由环境变量注入，不写入配置文件或代码仓库
- 密钥展示时强制脱敏，仅显示前 4 位 + 后 4 位（如 `AKID****xyz`）
- 支持登录失败锁定（5次失败锁定 30 分钟）
- 邮件/企业微信通知中不展示敏感信息

### 6.3 可扩展性

- 数据模型预留扩展字段（JSONField）以适应未来新增属性
- 拓扑图支持自定义节点样式和布局算法切换
- 备案变更审核流程支持自定义审批链（未来可扩展）
- 云平台适配器采用策略模式，新增云平台只需新增 Adapter 实现类，无需改动核心同步逻辑
- 告警通知渠道采用插件化设计，新增渠道（如钉钉、飞书）只需实现 `Notifier` 接口

### 6.4 云 API 专项要求

| 指标 | 要求 |
|---|---|
| SDK 依赖管理 | 各云平台 SDK 作为可选依赖，仅安装所需平台的 SDK |
| API 限流保护 | 遵守云平台 API 调用频率限制，内置请求间隔控制（默认 500ms/次） |
| 凭证失效处理 | 同步时若返回鉴权失败（AuthFailure），自动标记 sync_enabled=False 并触发告警 |
| 同步超时 | 单地域同步超时 5 分钟，单账号总体同步超时 30 分钟 |
| 数据一致性 | 同步过程中若发生异常，已同步的数据不回滚，仅标记 sync_log.status=partial |

## 八、里程碑规划建议

| 阶段 | 内容 | 产出 |

## 附录 A：术语表

| 术语 | 说明 |
|---|---|
| 资源节点 (Resource Node) | 拓扑图中的基本单元，代表一个项目/服务器/域名/IP/端口实体 |
| 关联链路 (Association Chain) | 从某一节点出发，沿关联关系追溯到的完整上下游节点序列 |

| 上游追溯 | 从域名/IP 向部署服务器和项目的方向追溯 |
| 下游追溯 | 从项目向服务器、IP、域名、端口的方向追溯 |
| 影响分析 (Impact Analysis) | 评估某资源状态变更对其他资源产生的影响范围 |
| 软删除 (Soft Delete) | 标记数据为已删除但不物理清除，保留数据可追溯性 |
| ICP 备案 | 互联网信息服务备案（Internet Content Provider filing），中国境内网站必须完成的合规手续 |
